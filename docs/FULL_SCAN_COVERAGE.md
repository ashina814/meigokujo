# 冥獄城全体把握 — 情報網羅性の定義

## 結論

Discordのメッセージ履歴を全件取得しても、冥獄城全体を理解するには不足する。

全体把握は、次の6系統を統合したときに初めて成立する。

1. Discordの現在構造と取得可能な過去ログ
2. 今後発生する変更イベントの継続記録
3. 冥獄城Botと関連BotのDB・設定・実装
4. 仕様書、運用文書、会議資料などの外部文書
5. ChatGPT上で整理済みの裁定・人物・運営情報
6. Discord APIで取得不能な事実に対する人間の訂正・補足

## 情報源別カバレッジ

| 情報源 | 基準スナップショット | 継続記録 | 主に分かること | 恒久的な限界 |
|---|---:|---:|---|---|
| Discordサーバー構造 | 必須 | 必須 | ロール、チャンネル、権限、オンボーディング、Bot構成 | 削除済み構造は監査ログ期限外だと復元不能 |
| Discordメッセージ | 必須 | 必須 | 発言、決定、依頼、報告、企画、議論 | 削除済み本文、閲覧権限外、DMは取得不能 |
| リアクション・投票 | 必須 | 必須 | 確認済みの人、参加意思、承認状況 | 過去に消された反応は復元不能 |
| スレッド・イベント参加者 | 必須 | 必須 | 案件関係者、担当者、参加者 | API権限とIntentに依存 |
| VC状態・BotのVC台帳 | 必須 | 必須 | 同席、滞在時間、活動時間帯 | 過去音声の内容は取得不能 |
| 監査ログ | 必須 | 必須 | 管理操作、ロール・チャンネル変更 | Discord側の保存は45日 |
| 冥獄城Bot DB | 必須 | 必須 | 経済、評価、入城、VC、チケット、イベント | Bot導入前・未記録の事実は存在しない |
| 関連Bot DB・コード | 必須 | 変更時 | 各Botが担う業務、設定、未実装領域 | サーバー外の保存先を別途特定する必要あり |
| OneDrive等の仕様書 | 必須 | 変更時 | 正式仕様、設計意図、業務フロー | Discord実運用との差を別途比較する必要あり |
| ChatGPT運営記録 | 必須 | 継続 | 裁定経緯、ヒアリング、暗黙知、分析 | 原文Discordへの根拠リンクがない情報もある |
| 人間による補足 | 必須 | 随時 | DM、通話、口頭決定、外部鯖での出来事 | 証言として扱い、確定事実と分離する |

## 基準スナップショットで必須のDiscord情報

### サーバーと導線

- Guild基本設定、Community設定、事故対策状態
- Welcome Screen
- Onboardingの質問、回答、付与チャンネル・ロール
- ルール・案内チャンネル
- Scheduled Eventと購読者
- Stage Instance

### 組織と権限

- 全ロール、順序、権限
- 全メンバー、現在ロール、参加日時、タイムアウト
- 全チャンネル、カテゴリ、権限上書き
- Bot自身が各場所をどこまで取得できたか
- 各メンバーの実効権限を再計算できる材料

### Bot・自動化

- Botユーザー一覧
- Guild Application Commands
- Webhook
- Integration
- AutoMod
- 招待
- Botと対応リポジトリ・DB・運用担当者の対応表

### 会話と案件

- 取得可能な全チャンネル・全スレッド
- archived private threadを含む取得可能な全スレッド
- スレッド参加者
- メッセージ本文、編集日時、返信・転送関係
- 添付ファイル実体、ハッシュ、元メッセージ
- Embed、Component、Sticker、Poll
- リアクションしたユーザー
- Pollの回答別投票者
- ピン留め

### 活動

- 現在のVC参加状態
- Bot DB内の過去VC区間
- メッセージ活動
- イベント参加
- 評価、昇格、入城、退城、Land、給与、チケット

## 単発スキャンでは分からないもの

次の情報は、基準スナップショットの後にイベントを継続記録しなければ分からない。

- メッセージの編集前本文
- 削除されたメッセージ本文
- リアクションの追加・解除履歴
- ロールの付与・剥奪履歴
- チャンネル・権限・スレッドの変更履歴
- 担当者が期限までに反応しなかった事実
- 案件が何日停止したか
- VCへの入退室と同席推移
- 人物・部署ごとの負荷変化

そのため、全体把握は次の二段構成にする。

```text
基準スナップショット
  +
以後の変更イベントを追記するイベントログ
```

## 完全性レポート

各領域に次の状態を付ける。

- `complete` — 期待件数と取得件数が一致し、必要権限も確認済み
- `partial` — 一部取得できたが、権限・Intent・上限・失敗がある
- `unavailable` — Discord APIまたは権限上取得不能
- `not_configured` — 取得処理をまだ実装していない
- `not_applicable` — サーバーで使われていない機能

最低限、以下をmanifestへ記録する。

- Guildの公称メンバー数と取得メンバー数
- チャンネル総数、閲覧可能数、履歴取得可能数
- スレッド総数の取得根拠とprivate thread権限
- メッセージ取得件数、最古・最新日時、途中失敗位置
- 添付総数、保存成功数、容量、失敗数
- リアクション総数とユーザー取得成功数
- 監査ログ最古日時
- DBテーブル数、行数、読み取り専用接続の確認
- IntentとBot権限

## 分析用の正規化モデル

生ログを直接AIへ渡すだけでは、冥獄城の現状にはならない。最低限、次へ変換する。

- `Person` — 人物、別名、ロール、所属、活動状態
- `RoleAssignment` — 役職の付与・剥奪と期間
- `Case` — 問題、裁定、関係者、状態
- `Project` — イベント・制度変更・Bot開発
- `Task` — 担当、期限、完了条件、状態
- `Decision` — 決定内容、決定者、適用範囲
- `Evidence` — 原文、添付、監査ログ、証言
- `Event` — 時系列上の出来事
- `VcSession` — VC滞在と同席
- `Metric` — 入城、定着、評価、イベント、経済等の集計

すべての分析結果は、元データのIDと取得元へ戻れるようにする。

## 情報の確度

- `verified` — Discordオブジェクト、監査ログ、DBなど機械的に確認
- `stated` — 本人または第三者の明示的発言
- `inferred` — 複数データからAIが推定
- `unknown` — 情報不足または矛盾

人物評価や処分判断は、`inferred`だけで確定しない。

## 完了条件

「全体スキャン完了」と呼べるのは、次を満たしたときだけ。

1. 本番DBを一切変更しないことが検証されている
2. CIの型チェックとテストが成功している
3. テストサーバーで全取得処理が完走している
4. 本番では最初に100件制限で試走している
5. 完全性レポートで重大領域が`not_configured`になっていない
6. 添付実体が保存され、期限付きURLだけに依存していない
7. スキャン結果の保存権限・暗号化・保持期間が決まっている
8. 基準スナップショット後の継続記録が開始されている
9. Discord以外の文書・Bot・ChatGPT情報の取り込み経路が決まっている
